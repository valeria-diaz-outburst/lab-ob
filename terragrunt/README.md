# Terragrunt — Lab Infrastructure

Terragrunt wrapper around Terraform for deploying Kubernetes services to the lab cluster. Designed to scale: adding a new service means adding a module and a single `terragrunt.hcl` file, with no repeated provider or backend configuration.

---

## Directory structure

```
terragrunt/
├── terragrunt.hcl                  # Root config — shared by all services
├── .gitignore                      # Excludes generated and state files
├── _modules/                       # Reusable Terraform modules (no Terragrunt here)
│   └── traefik/
│       ├── main.tf
│       ├── variables.tf
│       └── outputs.tf
└── lab/                            # Environment: lab
    ├── env.hcl                     # Environment-level variables
    └── services/                   # One subdirectory per deployed service
        └── traefik/
            └── terragrunt.hcl
```

---

## File-by-file breakdown

### `terragrunt.hcl` (root)

This is the most important file in the setup. Every service-level `terragrunt.hcl` calls `find_in_parent_folders()`, which walks up the directory tree until it finds this file and inherits everything in it. You define provider and backend config here once, and every service gets it for free.

**What it does:**

- **`locals` block** — reads `env.hcl` from the nearest parent directory. This lets the root config know which environment it's running in (e.g. `lab`) without hardcoding it.

- **`generate "provider"` block** — writes a `provider.tf` file into each service's working directory at `terragrunt init` time. It declares the `kubernetes` and `helm` providers, both pointing to `~/.kube/config` for cluster access. You never write `provider.tf` by hand in any module.

- **`remote_state` block** — configures a local Terraform backend and writes a `backend.tf` into each service's working directory. State files land at:
  ```
  terragrunt/.tfstate/<env_name>/<service_path>/terraform.tfstate
  ```
  Each service gets its own isolated state file. The path is derived automatically from the directory structure, so there are no conflicts between services.

### `.gitignore`

Excludes files that should never be committed:

| Pattern | Why |
|---|---|
| `.terragrunt-cache/` | Terragrunt downloads module source here; it's ephemeral |
| `provider.tf`, `backend.tf` | Generated by the root `terragrunt.hcl`; not hand-written |
| `.tfstate/` | Local state files; sensitive and environment-specific |
| `.terraform/` | Terraform provider binaries downloaded on `init` |
| `.terraform.lock.hcl` | Lock file; optionally commit this if you want pinned provider versions |

---

### `_modules/traefik/main.tf`

The actual Terraform resources for deploying Traefik. This is plain Terraform — no Terragrunt-specific syntax. It can be used standalone or called by Terragrunt.

**Resources:**

- **`kubernetes_namespace`** — creates the `traefik` namespace (or whatever `var.namespace` is set to) in the cluster before anything else is deployed.

- **`helm_release`** — deploys Traefik using its official Helm chart from `https://helm.traefik.io/traefik`. The `values` argument accepts arbitrary YAML, passed in from outside via `var.values_yaml`. The `depends_on` ensures the namespace exists before the Helm release is attempted.

---

### `_modules/traefik/variables.tf`

Declares the inputs the Traefik module accepts. Keeping variables here (separate from `main.tf`) makes the module's interface explicit and easy to find.

| Variable | Default | Purpose |
|---|---|---|
| `namespace` | `"traefik"` | Kubernetes namespace to deploy into |
| `chart_version` | `"28.0.0"` | Pinned Helm chart version — change this to upgrade Traefik |
| `values_yaml` | `""` | Helm values override as a raw YAML string |

Pinning `chart_version` is intentional. It prevents unexpected upgrades when you run `terragrunt apply` on a different day.

---

### `_modules/traefik/outputs.tf`

Exports values from the Traefik deployment that other modules or the operator might need.

| Output | Value |
|---|---|
| `namespace` | The namespace Traefik was deployed into |
| `release_name` | The Helm release name (`traefik`) |
| `release_status` | Helm release status (e.g. `deployed`) |

Outputs matter when services depend on each other. For example, a future `ingress-routes` module could reference `dependency.traefik.outputs.namespace`.

---

### `lab/env.hcl`

Defines variables scoped to the `lab` environment. The root `terragrunt.hcl` reads this file via `read_terragrunt_config(find_in_parent_folders("env.hcl"))`.

Currently it only sets `env_name = "lab"`, which is used to namespace state file paths. When you add more environments (e.g. `staging/`), each gets its own `env.hcl` with a different `env_name`, keeping state completely separate.

---

### `lab/services/traefik/terragrunt.hcl`

The service deployment config. This is what you run `terragrunt apply` from. It is intentionally thin — its only job is to wire the reusable module to environment-specific inputs.

**What it does:**

- **`include "root"`** — pulls in the root `terragrunt.hcl` (provider generation, backend config). This is what eliminates all boilerplate from service configs.

- **`terraform.source`** — points to the `_modules/traefik` directory. Terragrunt copies the module into `.terragrunt-cache/` and runs Terraform from there, keeping the module source clean.

- **`inputs`** — sets values for the module's variables. The `values_yaml` block here is a multiline YAML string passed directly to the Helm chart. This is where you tune Traefik behaviour (ports, service type, ingress class) without touching the module itself.

---

## Adding a new service

1. Create `_modules/<service>/` with `main.tf`, `variables.tf`, `outputs.tf`
2. Create `lab/services/<service>/terragrunt.hcl` pointing to that module
3. Run `terragrunt init && terragrunt apply` from the new service directory

The provider config, backend, and environment variables are inherited automatically. No copy-pasting.

---

## Common commands

```bash
# From any service directory (e.g. lab/services/traefik/)
terragrunt init        # download providers and set up backend
terragrunt plan        # preview changes
terragrunt apply       # deploy
terragrunt destroy     # tear down

# From lab/ — run across all services at once
terragrunt run-all plan
terragrunt run-all apply
```
